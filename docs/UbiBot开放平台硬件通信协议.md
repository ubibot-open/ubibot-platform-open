# UbiBot开放平台硬件通信协议

### 0. 项目定位与本次改动说明

本项目（ubibot-open）定位为一个内部使用的开源物联网平台实现，不是一个追求完整性、可靠性或生产级安全性的商用平台。首要目标是：让使用者能用尽量少的步骤、尽量少的配置，把一台硬件接入系统并看到数据——复杂的鉴权流程、状态机、下发通道，只要不是"接入数据"这个核心路径必需的，都尽量去掉。

相较于早期版本，本次改动：

- 面向内网可信环境的定位，设备身份鉴权也一并简化掉了：不再有 DeviceSecret、签名计算这些环节，设备只用 pid+sn 明文标识自己，不再需要提前在平台手动创建设备或走"自主激活审批/密钥绑定"流程——插上电、能联网，就能上报数据。
- 取消"激活"这个独立步骤（设备激活 / Token 会话），数据上传请求直接用 pid+sn 标识身份，一次请求即完成写入；保留了一个不做任何校验的轻量时间同步接口，方便没有本地时钟的设备获取当前时间。
- 取消指令下发通道，平台不再向设备推送 reboot / calibrate / set_probe / OTA 等控制指令。这类功能对"接入数据"这个核心目标不是必需的，教学/演示场景也很少需要。
- 上传数据不再使用 `temperature`、`humidity` 这类具名字段，统一改成 `field1` ~ `field20`（最多20个），平台只负责原样保存这些数值，不关心其业务含义。`field1`/`field2`/`field3` 三个字段有约定俗成的默认含义（温度/湿度/光照），其余由使用者自行定义。

以下正文按新的、简化后的协议描述。

### 1. 传输层

| 项 | 要求 |
| --- | --- |
| 协议 | HTTP |
| 方法 | POST |
| 数据格式 | JSON，UTF-8 |
| Content-Type | application/json |

注意：本协议不做任何身份鉴权，完全信任请求里的 pid+sn——只适用于可信的内网环境，不要把这个方案直接套用到需要真实安全性、或暴露在公网上的部署。

### 2. 设备身份

设备只需要两个字段标识自己，均明文传输，不需要密钥/签名：

| 字段 | 说明 |
| --- | --- |
| pid（ProductID） | 产品型号，用于区分设备类型 |
| sn（SerialNumber） | 设备唯一序列号 |

### 3. 时间同步

设备没有本地时钟（首次上电、RTC掉电重置）时，可以调用这个接口获取服务器当前时间；这个接口不做任何身份校验，不需要也不校验签名。

**POST /api/v1/auth/time**

请求：

```json
{ "pid": "ubibot_open_dev_v1", "sn": "sn_ws1_20001_1"}
```

响应：

```json
{ "c": 0, "t": 1788950400 }
```

| 字段 | 说明 |
| --- | --- |
| t | 服务器当前Unix秒时间戳 |

### 4. 数据上传（唯一的设备侧接口）

**POST /api/v1/data/report**

请求体：一次上报可携带多个时间点（例如设备离线期间缓存了多轮采集数据）：

```json
{
  "pid": "ubibot_open_dev_v1",
  "sn": "sn_ws1_20001_1",
  "ts": 1788950400,
  "payloads": [
    { "ts": 1788950400, "feed": { "field1": 25.6, "field2": 60.2 } },
    { "ts": 1788951000, "feed": { "field1": 25.8, "field2": 59.9, "field3": 812 } }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| pid | string | 是 | 产品型号 |
| sn | string | 是 | 设备序列号，识别设备身份的唯一依据（不做签名校验） |
| ts | Int64 | 是 | 本次请求发起时的Unix秒时间戳，用于基本的时间窗校验（可选校验，见§7） |
| payloads | array | 是 | 一条或多条记录，每条对应一个采样时间点 |
| payloads[].ts | Int64 | 是 | 该采样点的Unix秒时间戳（可以早于外层ts，用于补传离线期间缓存的数据） |
| payloads[].feed | object | 是 | `field1`~`field20` -> 数值 的映射，见§5。未使用的字段编号直接省略，不要传空值 |

响应：

```json
{ "c": 0, "t": 1788950400 }
```

| 字段 | 说明 |
| --- | --- |
| c | 业务状态码，0成功，其他值见§7 |
| t | 服务器当前Unix秒时间戳，设备可用它校准本地时钟（可选，非必须） |

服务端行为：
- 若 sn 从未出现过，自动创建该设备记录（pid/sn/首次上报时间等），后续按已存在设备处理。
- 若设备被后台管理员手动禁用，本接口拒绝该设备的所有数据（见§7，禁用是后台对现有设备的管理动作，不是新设备接入的前提条件）。

### 5. 数据字段（field1 ~ field20）

不再使用 `temperature`/`humidity` 等具名传感器字段，统一改为编号字段，最多20个。平台只按编号原样保存数值，不关心每个编号具体代表什么物理量——除了以下3个字段有约定俗成的默认含义：

| 字段 | 默认含义 |
| --- | --- |
| field1 | 温度 |
| field2 | 湿度 |
| field3 | 光照 |
| field4 ~ field20 | 由使用者自行定义（例如 CO2、土壤pH、电池电压等），平台不做任何假设 |

这个默认含义只是给后台展示（例如图标、单位）用的一个约定，不是协议层面的强制校验——设备完全可以只上报 field1，也可以从 field4 开始上报，平台都会照常保存。

### 6. 设备管理（后台侧，非设备侧协议）

以下内容不是设备需要实现的协议，而是后台管理的行为说明，帮助理解§4提到的"自动创建/禁用"：

- 设备首次成功上报数据时自动出现在后台设备列表里，无需预先创建。
- 管理员可以在后台给设备改名、查看历史数据、禁用/解除禁用、删除设备及其全部数据。
- 禁用一台设备后，该设备后续所有上报请求都会被拒绝（数据不再被处理），解除禁用后恢复正常。
- 删除设备会连同其历史数据一起永久删除。

### 7. 错误处理

| HTTP状态 | c | 场景 | 设备行为 |
| --- | --- | --- | --- |
| 200 | 0 | 成功 | 正常处理 |
| 400 | 1002 | 时间戳超出±5分钟窗口 | 检查本地时钟，或调用§3时间同步接口校准后重试 |
| 400 | 1003 | 请求体格式错误 | 检查固件序列化逻辑，下个周期重试 |
| 401 | 1103 | 设备已被后台禁用 | 停止重试并告警（LED/日志），需人工核实 |
| 429 | 1900 | 触发限流 | 放弃本次上报，等待下个周期 |
| 5xx | 5000 | 服务端故障 | 等待下个上传周期重试 |

### 附：本次移除的功能

为配合上面的定位调整，以下曾经存在于旧版协议里的能力本次被整体移除，不再是本项目范围：
- 设备身份鉴权（HMAC签名、DeviceSecret推导公式）——面向内网可信环境，进一步简化为仅用 pid+sn 明文标识设备，不做任何签名/密钥校验。
- 会话Token及其续期。
- 服务端向设备下发控制指令的整个通道（原`cmd`字段），包括配置轮询（原`/device/poll`）、探头自定义数据读取配置（原`set_probe`）、固件OTA升级。
- 自主激活设备的审批/密钥绑定流程（含加密提交、RSA公私钥）。
